# 从多仓到 Monorepo：一名前端工程师的工程化反思

> 草稿 · 已完成「背景介绍」「问题分析」两节，后续章节预留位见文末。  
> 作者立场：Vue 技术栈背景的前端工程师，从协作与工程化角度切入，而非工具评测。

---

## 一、背景介绍

### 1.1 前端工程的复杂度，正在从代码内部转移到代码之间

回望前端这十多年的发展，最显著的变化并不是某个框架的胜出，而是**项目数量与项目之间关系的爆炸**。

早些年，一个公司一个产品，一个产品一个仓库就够了。jQuery 时代，前端的复杂度主要在“一个 HTML 页面里能写得多复杂”。SPA 兴起之后，复杂度迅速向工程化迁移：模块化、构建工具、状态管理、组件抽象。在那个阶段，单仓库依然能很好地承载工作。

但今天的前端团队，面对的现实通常是这样：

- 一个业务线可能同时维护 PC 端、H5 端、小程序、运营后台、商家后台
- 一个产品里既有“面向用户的应用”，也有“面向研发的组件库 / SDK / 脚手架”
- 同一套设计语言要在多个应用里保持一致
- 多个项目共用同一份业务模型、接口契约、埋点规范

也就是说：**前端工程的复杂度，已经不再集中在“一个项目内部”，而是大量出现在“多个项目之间”**。代码本身的设计可以靠经验解决，但代码之间的协作方式，是组织层面的问题。

### 1.2 多仓（Multi-Repo）一度是默认答案

面对“多个项目”这一现实，业界最自然的反应是：**一个项目一个仓库**。

这种做法非常符合直觉：

- 边界清晰，权限好划分
- 仓库小、克隆快、CI 简单
- 团队各自负责自己的仓库，互不干扰
- 与微服务、康威定律的思路高度同构

为了在多个仓库之间共享代码，社区也很快形成了配套方案：发布私有 npm 包、维护内部 registry、用 `npm link` 或 `yalc` 联调。看起来一切都在朝着“专业化”的方向演进。

很长一段时间里，多仓被视为团队成熟度的体现。

### 1.3 直到问题开始浮出水面

但很多团队走着走着会发现一件事：**那些“专业化”的拆分，并没有让协作变得更轻松，反而引入了一种新的、更隐蔽的成本——协调成本。**

- 抽一个工具函数到公共库，要走完整的发布流程
- 改一个共享组件的 prop，要同步几个仓库
- 升级一个依赖，要在 N 个仓库重复几遍
- 每个仓库都有一份相似但不完全一致的 ESLint / tsconfig / CI 配置
- 新同学入职第一周，时间几乎都花在“克隆仓库、装环境、跑通联调”上

到了这个阶段，多仓不再是“清晰”的体现，而成了一种**结构性摩擦**：你能感觉到团队的速度被某种东西拖住了，但说不清是哪一行代码的问题。

### 1.4 Monorepo 进入主流视野

正是在这种背景下，Monorepo（单仓多项目）这个并不新的概念，重新被前端社区严肃讨论。

它最早被大规模实践的，是 Google、Meta、微软这种巨型工程组织。但真正让前端工程师产生共鸣的，是开源社区的转向：

- Babel、React、Vue、Vite、Vercel SDK、shadcn/ui ……  
  这些和我们日常打交道最多的项目，几乎都采用了 Monorepo 结构。

更重要的是，工具链开始变得对前端友好：

- `pnpm workspace` 让多包管理成本降到几乎为 0
- `Turborepo` / `Nx` 让任务编排和缓存变得开箱即用
- `Changesets` 让多包发布有了清晰的工作流

也就是说，今天讨论 Monorepo，已经不再是“要不要忍受复杂工具链”的问题，而是“**要不要重新审视团队的协作边界**”的问题。

这就是接下来这篇文章想讨论的事情。但在谈方案之前，我们必须先把多仓模式下的真实痛点拆开看清楚——因为如果不理解问题，我们换一种结构只是换一种姿势继续疼。

## 二、问题分析：多仓模式下，前端团队真正在为什么买单

下面这些问题，并不是多仓“无法解决”，而是多仓让它们**系统性地变贵**。理解这一点，比记住任何工具都重要。

### 2.1 代码复用：抽得起，养不起

理论上，公共代码应该被沉淀到独立仓库，发布成 npm 包，让大家共用。**实操中，这条路上的成本被严重低估。**

抽一个工具函数 `formatPrice`，看起来只需要几行代码，但真实流程是：

1. 新建一个仓库 / 在现有 utils 仓加一个文件
2. 写测试、补 README
3. 配置发布、版本号、CHANGELOG
4. 发布到私有 npm
5. 各业务仓库升级版本号
6. CI 跑通、走灰度、合并

更糟糕的是，公共库一旦发布出去，**它就有了维护责任**：bug 修复要发版，breaking change 要协调升级，废弃要走 deprecated 流程。

结果在实际工作中你会看到这样的现象：

- 大家心里都清楚某段逻辑应该抽出来，但**没人愿意先动手**
- 抽出来的公共库迅速沦为“能跑就行，没人优化”的状态
- 复制粘贴的代码反而成了“低成本”选项

**问题本质**：复用的边际成本太高，导致复用收益被挤压成负数。

### 2.2 跨项目改动：协调地狱

公共库一旦被多个项目消费，任何**有意义的改动**都会变成一场跨仓 PR 接力。

举个真实场景：你想把 `Button` 组件的默认 size 从 `medium` 改成 `small`，因为新设计规范要求这样。流程是：

1. 在组件库仓库改代码、发布新版本
2. 通知所有消费方升级
3. 每个消费方在自己仓库提 PR、改用法、做回归
4. 等所有 PR 合并、所有应用部署完
5. 才算这个改动“真正完成”

这条路径上有几个固有问题：

- 链路长：一个改动要走 N 条独立流水线
- 不可见：每个仓库的进度只在自己 PR 里，发起方很难全局追踪
- 不可回滚：某个消费方升级失败时，公共库无法回退（其他人已经升上去了）
- 心智沉重：发起人会本能地避免这种改动，于是公共库越来越僵化

**问题本质**：物理上分散的仓库，无法在一次原子操作里完成跨仓修改。

### 2.3 版本漂移：你不知道你在用哪个版本

多仓 + 私有 npm 的组合，必然会带来版本碎片化：

- 项目 A 用的是 `@company/ui@1.2`
- 项目 B 用的是 `@company/ui@1.5`
- 项目 C 用的是 `@company/ui@2.0-beta`

接下来发生什么？

- 一个 bug 在 A 仓被报告，但只在某些版本存在
- 复盘时无人能立刻说清楚“当时究竟是哪个版本”
- 升级 A 到 2.0，发现之前 1.x 的某个隐式行为被依赖了
- 老版本要不要再发个 patch？谁来维护？

更隐蔽的是**间接依赖的漂移**：你直接依赖的库都对得上，但它们各自依赖的子依赖不一致，最终在运行时表现出微妙差异。这种问题排查起来极其消耗心力。

**问题本质**：缺少一个能让“全仓库视角下，所有项目此刻在用什么”一目了然的全局视图。

### 2.4 工程化规范：写得出来，守不住

每个新仓库刚建立时，往往会拷贝一份比较完善的 ESLint、Prettier、tsconfig、CI 模板。前两个月一切正常，半年后再看，往往是这样：

- 仓库 A：还停留在最初的 ESLint 8.x，有一堆历史豁免规则
- 仓库 B：升级到了 ESLint 9，但漏了 import 排序规则
- 仓库 C：tsconfig 的 `strict` 被关了，没人记得为什么
- 仓库 D：CI 的 Node 版本和别人差一个大版本

每个仓库单独看都没什么问题，但**横着比，就是一片混乱**。

更糟的是：当你想统一升级（比如全公司切到 ESLint flat config），你面对的不是“升级一次”，而是“升级 N 次，每次都不太一样”。

**问题本质**：规范一旦被复制成 N 份，就不再是规范，而是 N 个独立演化的副本。

### 2.5 本地联调：HMR 断裂、效率退化

这是最容易被低估的痛点。

假设你在做一个功能：组件库里要加一个新的 `Drawer` 组件，主应用要用上它。多仓模式下的开发节奏可能是：

1. 改组件库代码
2. 本地构建
3. `npm link` 或 `yalc add` 到主应用
4. 主应用重启 dev server
5. 调试，发现 bug，回到第 1 步

这中间最致命的是：

- `npm link` 经常因为路径、依赖去重等问题出诡异 bug
- 每次组件库改动，主应用 HMR 要么不生效，要么需要重启
- 多人协作时，A 的本地组件库版本和 B 的对不上

久而久之，团队会形成一种隐性习惯：**“能不在主应用里联调就不联调，先发个 alpha 版本上去再说”**。这看似省事，实际上把质量保证的责任甩给了部署后的环境。

**问题本质**：物理上隔离的代码，在开发期被强行用工具连接，必然脆弱。

### 2.6 CI/CD：每个仓库重复造轮子

多仓的 CI 体系往往呈现这样的状态：

- 每个仓库一份近似但不完全一样的流水线 YAML
- 每个流水线都装一遍同样的依赖
- 每个流水线都跑一次相似的 lint / test / build
- 工具升级时要在 N 个仓库各改一遍

更深一层的问题是：**没法做“跨仓库一致性测试”**。

比如你升级了组件库，理论上应该跑一遍“所有消费方的回归测试”。但在多仓体系里，组件库 CI 看不到消费方，消费方 CI 看不到上游变化，整体测试只能靠人脑串起来。

**问题本质**：CI 的边界等于仓库边界，但变更的影响范围常常跨仓库。

### 2.7 协作信号：散落在多个仓库的 PR，等于没有信号

工程师每天有大量决策依赖于“**别人最近在做什么**”这个信号：

- 我要改这个文件，最近有人动过吗？
- 这个 API 是不是有人正在重构？
- 这个新功能是哪条业务线在推？

多仓模式下，PR、issue、commit 散落在 N 个仓库里。除非你刻意去看，否则你只能看到**自己仓库里发生的事**。一个产品的全貌，没人能从代码层面拼出来。

时间一长，这种割裂会演化成更严重的问题：

- 重复造轮子（不知道别人已经做过类似的事）
- 设计漂移（不同仓库的同类组件越长越不像）
- 文化稀释（review 标准、命名风格、提交规范分裂）

**问题本质**：协作是基于可见性的，而多仓默认就是把可见性切碎。

### 2.8 知识与历史：跨仓的事，没人记得

最后一个问题，最少被讨论，但最致命。

一个产品的设计决策，往往是横跨多个仓库的：

- “为什么 Button 的 disabled 状态用透明度而不是灰色？” —— 在 UI 仓库的某个 PR 里
- “为什么这个 API 被废弃了？” —— 在网关仓库的某次讨论里
- “为什么我们不用全局 store？” —— 在主应用的早期 commit 里

多仓模式下，这些线索散落在不同仓库的 git 历史和 PR 描述中。**没有任何一个地方，能让一个新人通过翻阅 git 历史，理解这个产品是怎么演化到今天的。**

久而久之，团队的“工程记忆”会被仓库边界切割成无数碎片。新人难以建立全局认知，老人离职后的知识断层难以修复。

**问题本质**：代码的历史是组织最宝贵的资产之一，多仓让这份资产无法被整体阅读。

---

### 小结：这些问题的共同根源

把上面 8 个问题放在一起看，你会发现它们指向同一件事：

> **代码的物理边界（仓库）和协作的逻辑边界（产品 / 团队 / 演进单元），并不重合。**

多仓让物理边界变得清晰，但代价是把本应连续的协作切成碎片。  
工具可以缓解症状（私有 npm、yalc、跨仓 CI 平台等），但只要边界错位的本质不变，团队就会持续为它付出隐性成本。

这正是 Monorepo 之所以在今天被认真讨论的原因——**它不是要“合并仓库”，而是要让代码边界回归到协作边界**。

## 三、Monorepo 是怎么解决这些问题的：机制层面，不只是工具

很多文章会把 Monorepo 的优点写成一份功能清单：“代码共享、统一构建、缓存加速……”。这种写法的问题是：**工具特性容易被记住，但解决问题的机制容易被忽略**。

下面我把第二节的 8 个痛点和 Monorepo 的机制一一对应起来——你会发现，它解决问题的核心，并不是某个具体功能，而是**让物理结构匹配协作结构**之后，许多问题自然消失了。

### 3.1 物理统一带来“原子变更”

在多仓里，跨仓改动是若干次独立提交、独立 PR、独立部署的串联。在 Monorepo 里，**同一个 PR 可以同时修改组件库和所有消费方**。

具体收益：

- 改一个共享组件 API，组件代码、所有消费方调用、相关测试，全部在一次 commit 完成
- Reviewer 在一个 diff 视图里就能看到“改动的全貌”
- 出问题时回滚一次 commit 即可，不用追多个仓库的回滚顺序

这一点直接消解了 [2.2 协调地狱]。共享代码维护的心理成本骤降，**“我应该重构”和“我能重构”之间的距离被拉近**。

### 3.2 源码级消费 + workspace 协议消除联调断层

Monorepo 内部包通常以源码方式被消费（包入口直接指向 `src/index.ts`），加上 pnpm 的 `workspace:*` 协议，主应用 import 内部包时，看到的就是源码本身。

带来的改变：

- 改组件库代码，主应用 HMR **立即生效**，不需要构建、不需要发版
- 不再需要 `npm link` / `yalc`，那一类“临时桥”彻底退出工作流
- 多人协作时不存在“你本地组件库版本和我的不一致”

直接消解 [2.5 本地联调]。**联调从“工具技巧”变回“纯写代码”**。

### 3.3 单一锁文件让“版本”这个变量消失

整个 Monorepo 共用一份 `pnpm-lock.yaml`。所有内部包之间互相引用走 `workspace:*`，没有版本号；外部依赖只在锁文件里固化一次。

含义是：

- 不存在“项目 A 用 1.2，项目 B 用 1.5”这种现象——大家用的是**同一份代码本身**
- bug 复盘不再需要先对齐版本号
- 升级外部依赖是一个全局动作，不是 N 个独立动作

[2.3 版本漂移] 在这个机制下基本失效。**你看到的版本就是真实的版本**。

### 3.4 共享配置包让“规范”从复制变成继承

Monorepo 里非常自然的一个模式是：把 `tsconfig`、`eslint-config`、`prettier` 等也作为 package 沉淀。其他包通过依赖+继承的方式使用，而不是各自拷贝一份。

```jsonc
// packages/web/tsconfig.json
{
  "extends": "@repo/tsconfig/vue.json",
}
```

```js
// packages/web/eslint.config.js
import base from "@repo/eslint-config/vue";
export default base;
```

带来的改变：

- 升级规则只需要改一个地方，所有包同步生效
- “这个仓库 lint 规则不一样”这种情况从根上消失
- 新建包成本极低：拷贝 5 行 `package.json` + extends 配置即可

[2.4 工程化规范守不住] 被结构性消解。**规范不再是 N 份副本，而是一个唯一可信源**。

### 3.5 统一 CI 视图 + 增量执行解决重复成本

Monorepo 的 CI 体系有两个关键变化：

1. **一份流水线管所有包**：不再是 N 份近似的 YAML
2. **affected 检测**：CI 通过 git diff 推算出本次变更影响了哪些包，只跑必要的任务

举个例子：你只改了 `packages/utils`，CI 的判断是：

- 重新 lint/test：`utils` 自己 + 所有依赖 `utils` 的包
- 跳过：完全不依赖 `utils` 的包

这种增量策略让大型 Monorepo 也能保持 CI 时间可控。同时也让“**跨仓一致性测试**”变成默认行为：升级共享组件后，所有消费方的回归测试自动跑一遍，不需要人工协调。

直接消解 [2.6 CI 重复成本]。

### 3.6 协作信号集中，让团队“看见彼此”

所有 PR、issue、commit 都汇聚在一个仓库里。每天扫一眼 PR 列表，你就能感知到：

- 哪些功能在被推进
- 哪个共享组件最近被改动得很频繁
- 谁在重构哪一块

这个看似简单的变化，对团队文化影响极大。它把“**别人最近在做什么**”这条信号从“需要主动打听”变成“被动可见”。

[2.7 协作信号被稀释] 在物理层面被解决。

### 3.7 共仓历史让“工程记忆”可被阅读

整个产品的代码历史在同一个 git 仓库中，意味着：

- 一次 PR 可以同时记录组件、应用、文档的演变
- 跨包的设计决策有完整上下文
- 新人通过 `git log` 就能追溯一个特性是怎么演化到今天的

这是 [2.8 知识割裂] 唯一可靠的解药——**让历史回到一个可被整体阅读的容器里**。

### 3.8 复用门槛降到最低

最后回头看 [2.1 代码复用]。在 Monorepo 里，抽一个工具函数到公共包的成本是什么？

- 在 `packages/utils/src` 加一个文件
- 在 `index.ts` re-export
- 主应用直接 import

不需要发版、不需要版本号、不需要 CHANGELOG、不需要协调升级。**复用的边际成本接近 0**，于是“该不该抽出来”这个问题，回到了纯粹的代码设计层面，而不是工程协作博弈。

### 3.9 把 8 条线索拧成一句话

如果要把这些机制总结成一句话：

> **Monorepo 不是“合并仓库”，而是让代码的物理边界回归到协作的逻辑边界。**

它解决的不是某一个具体痛点，而是“**边界错位**”这个根因。所有 8 个问题之所以会消失，是因为产生这些问题的结构性条件不再存在。

但这并不意味着 Monorepo 是免费的——它把一类问题消解的同时，引入了另一类问题。下一节我们就来看那一面。

---

## 四、Monorepo 不是银弹：它带来的新挑战

只讲优点的文章不可信。Monorepo 真正落地之后，团队会遇到一些“新”的、需要严肃对待的问题。理解这些代价，才能判断它是否真的适合你。

### 4.1 仓库膨胀：克隆、CI、IDE 都会变慢

包多了之后，最直接的代价是物理体积：

- 仓库 clone 时间从秒级变成分钟级
- IDE 索引明显变慢，全局搜索体验下降
- CI 镜像构建、缓存还原成本都上升

应对方式：

- 启用 Git 的 partial clone / sparse checkout（按需拉取）
- 给 IDE 配置 workspace 范围（只索引当前在做的子目录）
- CI 上利用 Turborepo / Nx 的 affected + 缓存避免全量执行

但要承认：**单仓的“轻量感”一去不返**。这是结构选择带来的固有成本。

### 4.2 边界模糊：所有人都能改所有代码

多仓时代，仓库本身就是权限边界。Monorepo 拆掉这堵墙之后，理论上每个人都能改任何代码。

这件事看似自由，实际上会带来：

- **责任稀释**：没人觉得某个 package 是“自己的”，bug 修复推诿
- **风格漂移**：不同 contributor 在同一个包里留下不同口味的代码
- **意外耦合**：业务代码偷偷 import 内部基础包的私有实现

应对方式不是回到多仓，而是**显式补回边界**：

- 通过 CODEOWNERS 文件明确每个包的负责人
- 通过 ESLint 规则禁止跨包的非公开 import（如 `import` 子路径限制）
- 通过包的 `exports` 字段精确控制对外 API

边界不再由“仓库”天然提供，而是要由“规则”显式表达。

### 4.3 工具链复杂度的跃迁点

当 package 数量超过 20，团队就会遇到一个临界点：

- 简单的 pnpm workspace 不够用了，开始考虑 Turborepo
- 缓存命中率开始变成需要监控的指标
- CI 时间长到必须做 affected 优化
- 内部包的 API 演进开始需要类似 Changesets 的工具

到这个阶段，Monorepo 的工具链本身会变成一个**专门的工程方向**——你需要有人愿意维护它，否则迟早“被工具反噬”。

直接的判断指标：**当“维护 Monorepo 工具链”每周占用一名工程师超过 1 天，你就需要一个明确的负责人。**

### 4.4 “大泥球”风险被放大

一个反直觉的事实：**边界不清晰的团队，Monorepo 比多仓更危险**。

多仓时代，仓库墙强制让你想清楚“这块代码归谁”。Monorepo 把墙拆了之后，如果团队没有边界纪律，结果就是：

- `packages/shared` 变成什么都往里塞的“神奇文件夹”
- 业务包之间循环依赖
- “共享”代码越长越像“万能胶水”

判断你是不是在走向这种状态的简单标志：

- 一个 PR 改了 5 个以上互不相关的 package
- `shared` / `common` / `utils` 这种包占代码量超过 30%
- 你说不清某个 package 的边界规则

如果这些信号出现了，**问题不是 Monorepo，是你的领域建模**。Monorepo 只是把这件事放大了。

### 4.5 CI 单点风险

所有项目共用一条流水线，意味着：

- 一处 lint 报错，可能阻塞整个组织的合并
- 工具升级一旦失误，影响所有人
- 流水线挂掉，整个团队的发布节奏受影响

这是中心化系统固有的代价。应对手段是把流水线本身当成一个“**生产服务**”来对待：版本化、灰度、可回滚、有监控。

### 4.6 新人冲击：第一周的“被淹没感”

最后一个常被低估的点：**新人第一次打开一个 80 个包的 Monorepo，会本能性懵掉**。

- 应该从哪个包开始读？
- `@repo/foo` 和 `@repo/foo-core` 有什么区别？
- 我要改的功能涉及哪些包？

应对方式：

- 写一份**面向新人的仓库地图**（README + 包关系图）
- 提供 task 级入口（`pnpm dev:web` 而不是让新人理解整个 turbo 配置）
- Code review 时主动把跨包链路讲清楚

这不是 Monorepo 独有的问题，但 Monorepo 让它显得更显眼。

### 4.7 给一个清醒的结论

Monorepo 解决的是“**协作边界**”问题，不解决“**领域设计**”问题。

如果团队在多仓时代就有清晰的领域划分、健康的协作文化，Monorepo 会让这一切跑得更顺；如果原本就没有，Monorepo 只会把混乱以更高密度的方式呈现出来。

**它是一面放大镜，不是一剂良药。**

---

## 五、前端 Monorepo 技术方案选型

工具层面，前端 Monorepo 真正需要做决策的就 4 个角色：

| 角色       | 解决什么问题                     | 主流候选                       |
| ---------- | -------------------------------- | ------------------------------ |
| 包管理     | 依赖安装、workspace 链接、锁文件 | pnpm（首选）/ yarn berry / npm |
| 任务编排   | 跨包构建顺序、并行、缓存         | Turborepo / Nx                 |
| 版本发布   | 多包版本号管理、CHANGELOG        | Changesets / Nx release        |
| 大规模治理 | 代码生成、依赖图分析、规则强制   | Nx / Rush                      |

下面只讨论关键决策点，不重复工具特性。

### 5.1 包管理为什么默认选 pnpm

三个核心原因：

1. **workspace 一等公民**：`workspace:*` 协议、`--filter` 过滤都是 pnpm 原生设计，体验最自然
2. **磁盘和安装效率**：依赖通过 content-addressable store 共享，单台开发机省下大量空间
3. **严格依赖隔离**：默认不会让 A 包用上没声明的依赖（“幽灵依赖”问题），这在 Monorepo 里非常重要

yarn berry 也能做，但配置成本更高、生态略小。npm workspace 在小型项目可用，大型项目缺乏过滤和性能优势。

**结论**：除非有特别理由，pnpm 是默认选择。

### 5.2 任务编排：Turborepo vs Nx

这两个的真实区别不是“谁更强”，而是“**面向哪一种团队**”。

| 维度         | Turborepo                  | Nx                                    |
| ------------ | -------------------------- | ------------------------------------- |
| 哲学         | 轻量、约定优于配置         | 平台化、强治理能力                    |
| 学习曲线     | 平缓，1 天能用起来         | 较陡，需要理解 generators / executors |
| 配置         | 一个 `turbo.json` 即可     | 多文件、多概念，但更灵活              |
| 代码生成     | 不内置                     | 内置，可用模板快速建包                |
| 适用规模     | 5–50 包                    | 50+ 包，多团队协作                    |
| 适合团队画像 | 中小前端团队、独立开源项目 | 大型组织、跨团队前端平台              |

经验性建议：

- 如果你是“**业务前端团队 + 一两个共享包**”：选 Turborepo
- 如果你是“**前端基建团队 + 多业务线 + 强治理诉求**”：选 Nx
- 如果你只有 3 个包：**先不上任何编排工具**，等真感到痛再加

### 5.3 Rush 适用范围其实非常窄

Rush 来自微软，强项是**大规模多包发布治理**（数百个包、严格的 publish 流程）。它的代价是：

- 配置非常重（`rush.json` + 多份配置）
- 与社区主流工作流（Vite、Vitest 等）整合需要额外工作

**如果你不是在做超大规模、对外发布大量 npm 包的基建项目，可以基本忽略它。**

### 5.4 版本发布：Changesets 是默认

只有“需要对外发布 npm 包”时才考虑这个角色。Changesets 的好处：

- 一次提交一个 changeset 文件，记录“改了什么、属于什么级别（patch/minor/major）”
- 发布时自动算版本、自动生成 CHANGELOG
- 不强制和具体编排工具绑定

如果你只是公司内部用，所有包都是 `private: true`，可以暂时不引入。

### 5.5 选型决策框架（可以直接套用）

按下面顺序问自己：

1. **包数量是不是在 5 个以内？**  
   是 → `pnpm workspace` 一个工具就够，别上任何编排。

2. **是否要对外发布 npm 包？**  
   是 → 加 `Changesets`。  
   否 → 跳过。

3. **CI 时间是不是开始变慢，或经常重复跑没必要的任务？**  
   是 → 加 `Turborepo`。  
   否 → 等真痛了再说。

4. **是不是多个团队在共用这个 Monorepo，且需要代码生成、依赖图治理？**  
   是 → 评估迁移到 `Nx`。  
   否 → 留在 Turborepo。

5. **是不是要管理上百个对外发布的 npm 包？**  
   是 → 评估 `Rush`。  
   否 → 不要碰它。

**核心原则**：工具数量永远控制在“当前真实需要”的最小集，避免被工具复杂度反噬。

## 六、最小可用 Monorepo 实战（Vue + pnpm workspace）

讲清楚机制和选型之后，下面用最少的代码把骨架搭起来。这一节是给读者一个“可以照抄复现”的最小样本。

### 6.1 目录结构

```
monorepo-demo/
├── apps/
│   └── web/                # Vue 主应用
└── packages/
    ├── ui/                 # Vue 组件包
    └── utils/              # 通用工具包
```

约定：`apps/*` 是“可独立运行的产品”，`packages/*` 是“被复用的库”。

### 6.2 三段关键配置

声明 workspace（`pnpm-workspace.yaml`）：

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

内部包入口指向源码（以 `packages/utils/package.json` 为例）：

```json
{
  "name": "@repo/utils",
  "private": true,
  "type": "module",
  "exports": {
    ".": {
      "types": "./src/index.ts",
      "import": "./src/index.ts"
    }
  }
}
```

主应用通过 workspace 协议引用（`apps/web/package.json`）：

```json
{
  "dependencies": {
    "vue": "^3.5.0",
    "@repo/ui": "workspace:*",
    "@repo/utils": "workspace:*"
  }
}
```

这三段就构成了最小 Monorepo 的全部魔法。

### 6.3 三个不能跳过的设计决策

这三件事在前面章节都有解释，这里只做提示式总结：

1. **包入口指向源码**：让主应用 Vite 直接消费 TS/Vue 源码，HMR 立即生效；代价是不能用于对外发布
2. **Vue 用 peerDependencies**：组件包不自带 Vue，由消费方提供，保证全仓只有一份 Vue 实例
3. **单向依赖图**：`web → ui → utils`，永远不要让 `utils` 反向依赖 `ui`

### 6.4 端到端工作流

```bash
# 第一次
pnpm install            # 一次安装全部包

# 日常开发
pnpm dev                # 启动主应用，HMR 联动所有内部包
pnpm build              # 按依赖顺序构建所有内容

# 新增内部包
mkdir packages/new-pkg
cd packages/new-pkg
pnpm init
# 修改 package.json 的 name 为 @repo/new-pkg
# 在主应用中：pnpm --filter web add @repo/new-pkg
```

跑通这套流程，你就拥有一个**真正可用的最小 Monorepo**。剩下的内容都是在这个骨架上叠加“协作能力”和“规模化能力”。

## 七、从“能跑”到“好协作”：工程化标准化

最小骨架跑通之后，Monorepo 的真正价值要靠工程化标准化释放。这一节讲三件最值得做的事：**共享 TS 配置、共享 Lint 配置、共享测试约定**。

### 7.1 共享 TypeScript 配置

新建 `packages/tsconfig`，里面放一份基础配置和若干变体：

```jsonc
// packages/tsconfig/base.json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "noUncheckedIndexedAccess": true,
  },
}
```

```jsonc
// packages/tsconfig/vue.json
{
  "extends": "./base.json",
  "compilerOptions": {
    "jsx": "preserve",
    "lib": ["ES2022", "DOM"],
  },
}
```

各包消费方式：

```jsonc
// packages/ui/tsconfig.json
{
  "extends": "@repo/tsconfig/vue.json",
  "include": ["src"],
}
```

收益：

- 所有 TS 配置升级，只需要改 `base.json` 一处
- 新建包的 `tsconfig` 永远只有 3 行
- 想做差异化（比如让 `apps/*` 比 `packages/*` 更宽松）就增加一个变体即可

### 7.2 共享 ESLint 配置

新建 `packages/eslint-config`，导出一份预设：

```js
// packages/eslint-config/vue.js
import vue from "eslint-plugin-vue";
import tseslint from "typescript-eslint";

export default [
  ...tseslint.configs.recommended,
  ...vue.configs["flat/recommended"],
  {
    rules: {
      "vue/multi-word-component-names": "off",
    },
  },
];
```

各包通过自己的 `eslint.config.js` 继承：

```js
import vueConfig from "@repo/eslint-config/vue";
export default vueConfig;
```

这一步彻底消除了 [2.4 工程化规范守不住] 的根本条件——规范不再是 N 份独立副本。

### 7.3 测试约定：Vitest 工厂

测试本身写在每个包内部，但**配置**应该收敛。可以在共享配置包里导出一个 Vitest 工厂：

```ts
// packages/vitest-preset/vue.ts
import { defineConfig, type UserConfig } from "vitest/config";
import vue from "@vitejs/plugin-vue";

export function createVueVitestConfig(overrides?: UserConfig): UserConfig {
  return defineConfig({
    plugins: [vue()],
    test: {
      environment: "jsdom",
      globals: true,
      ...overrides?.test,
    },
    ...overrides,
  });
}
```

每个包：

```ts
// packages/ui/vitest.config.ts
import { createVueVitestConfig } from "@repo/vitest-preset/vue";
export default createVueVitestConfig();
```

到这一步，新建一个 Vue 内部包的全部成本是：

- 一个 `package.json`（5 行）
- `tsconfig.json` 继承共享配置（3 行）
- `eslint.config.js` 继承共享配置（2 行）
- `vitest.config.ts` 调用工厂（2 行）

**新建包的成本越低，团队就越愿意拆出合理的小包**——这是健康 Monorepo 的正向循环。

### 7.4 提交侧的强制保障

最后一层守护是提交侧：

- `husky` + `lint-staged`：commit 前自动 lint 改动文件
- `commitlint`：约束提交信息风格
- CI 上跑一遍完整 lint + test

这些工具的细节这里不展开，关键是**把规范从"建议"变成"必经流程"**。Monorepo 拆掉了仓库墙，但需要在工作流里补回纪律。

## 八、规模化之后：缓存、affected、远端协作

当包数量超过 10 个、CI 时间开始让人忍受不了，就到了 Monorepo 的“规模化阶段”。这一节讲三个关键能力。

### 8.1 任务缓存：Monorepo 最值钱的能力

引入 Turborepo 之后，每次 `turbo run build` 实际经历的是：

1. 根据**输入文件 + 依赖 + 配置**计算出一个 hash
2. 查缓存：命中 → 直接复用结果（毫秒级）；未命中 → 真实跑任务，结果存入缓存
3. 命中和未命中的产物对 CI 完全等价

举一个真实数字感的例子：

- 一个 30 包的 Monorepo，全量 build 6 分钟
- 改 1 个叶子包，受影响的只有 4 个包
- 启用缓存后，CI 时间降到 40 秒

**缓存命中率在大型 Monorepo 里是一个值得监控的指标**，命中率低通常意味着任务输入定义不准确。

### 8.2 Affected：只跑必要的事

`affected` 的核心思想是：**根据 git diff 算出本次变更影响了哪些包**，然后只对这些包执行任务。

CI 上典型用法：

```bash
turbo run lint test build --filter=...[origin/main]
```

含义是：从 main 分支到当前分支，所有变更影响的包，跑完 lint/test/build。

这一步把 [2.6 CI 重复成本] 在大规模下也能控制住。**规模越大，affected 的收益越大。**

### 8.3 远端缓存：把效率延伸到团队

Turborepo / Nx 都支持把缓存上传到远端服务（Vercel 提供 / 自建 / Nx Cloud）。带来的改变：

- A 同学构建过的版本，B 同学拉代码后**直接命中**，不用本地再 build
- CI 跑过的产物，开发机可以复用
- 整个团队共享一份“知识库”形态的构建缓存

经验值：远端缓存对**多人 + 多机器**的团队收益巨大；对单人项目几乎没意义。

### 8.4 多包发布：Changesets 工作流

如果有对外包，发布要解决三个问题：版本号怎么涨、CHANGELOG 怎么写、发布顺序怎么定。Changesets 的工作流是：

1. 开发者在 PR 里运行 `pnpm changeset`，写一段“改了什么 + 升级类型”
2. 这个 changeset 文件作为 PR 的一部分被合入 main
3. 一个常驻 PR（“Version Packages”）会自动累积所有 changeset，并在合并时执行：bump 版本、更新 CHANGELOG、发布到 npm

这个流程的好处是：**版本和 CHANGELOG 是 PR 自然产物，不需要发布日开会决定**。

### 8.5 一条规模化的演进路径

总结成一条路径：

1. **5 包以内**：只用 pnpm workspace
2. **5–15 包**：加入 Turborepo（解决重复任务）
3. **有对外发布**：加入 Changesets
4. **15+ 包 + 多人协作**：开启远端缓存
5. **30+ 包 + 多团队**：评估迁移到 Nx 或继续深化 Turborepo 治理

**不要跳级**。每个阶段都先跑通、感受到收益，再进入下一阶段。

## 九、什么时候不该用 Monorepo

文章如果只讲“什么时候用”，是不完整的。Monorepo 不是普遍适用，下面这些情况，多仓往往是更好的选择。

### 9.1 项目规模真的很小

如果你只有一个主应用 + 一个工具库，并且未来一年内可预见不会扩展，**就别 Monorepo 了**。直接两个仓库，加一个发布到私有 npm，反而更简单。

判断标准：包数量 ≤ 2，且彼此变更频率不一致。

### 9.2 代码主权和合规要求不一致

举例：

- 一部分代码要开源，一部分代码必须闭源
- 一部分代码服务于客户 A，包含其敏感配置
- 不同子项目需要不同的访问权限

这种场景，**仓库本身就是合规边界**，不该被合并。

### 9.3 团队边界与代码边界分裂严重

Monorepo 假设“一个仓库 = 一群高频协作的人”。如果你把两个**几乎不交流**的团队的代码合到一起，他们会发现：

- 协作并没有变好
- 但 CI 时间、代码冲突、规范争论都增加了

**Monorepo 是协作工具，不是组织工具。**它无法替你解决组织割裂。

### 9.4 工程化基础不够

如果团队连以下几件事都还没稳定做到：

- 有清晰的代码规范并被执行
- 有像样的 CI（lint / test / build）
- 有基本的测试覆盖
- 有人愿意维护工具链

那么 Monorepo 反而会让事情更糟。**先把工程化基础修好，再上 Monorepo。**

### 9.5 工具链能力跟不上规模

包数量超过几十、构建时间已经成问题，但团队又没人愿意学 Turborepo / Nx 的高级特性，结果就是“**Monorepo 体积上去了，但效率没上去**”。

这时候你需要做的不是怪 Monorepo，而是**给工具链找个真正的负责人**。如果没有，先停下来，不要继续扩张。

### 9.6 一句话判断

> **如果你的团队具备清晰的领域划分、健康的协作文化、稳定的工程化基础，Monorepo 是放大器；否则，它是放大镜——会把所有问题以更刺眼的方式呈现出来。**

---

## 十、结语：工程化是协作问题，不是工具问题

回到这篇文章最初的论点：

- 多仓时代的痛苦，本质上是**代码物理边界与协作逻辑边界错位**带来的
- Monorepo 之所以有效，是因为它让两条边界重新对齐，而不是因为它“多了什么功能”
- 工具（pnpm、turbo、nx、changesets）只是把这种结构变得可维护，不是结构本身

所以，对一名前端工程师，特别是已经在团队里承担一定工程化责任的同学，我的建议是：

1. **不要为了 Monorepo 而 Monorepo**。先看你团队真正在为什么买单——是协调成本？版本漂移？联调断裂？还是全部都有？
2. **从最小骨架开始**：pnpm workspace + 一个应用 + 两个包。先体验机制，再考虑工具叠加。
3. **始终把"边界"看得比工具更重**。不管选 Turborepo 还是 Nx，包的边界、依赖的方向、对外 API 的稳定性，永远是更重要的事。
4. **把规范沉淀成 package**：tsconfig、eslint-config、test preset。这是 Monorepo 区别于多仓的最大杠杆。
5. **接受 Monorepo 不是终点**：当团队规模、产品复杂度变化时，结构也应该跟着演化。今天的最优解，未必是明年的。

工程化最终要解决的从来不是“代码怎么放”，而是“**人怎么一起把代码做好**”。Monorepo 只是给这个目标提供了一个更顺手的容器。容器之外，工程师的判断力、设计感、协作默契，才是真正决定项目能走多远的东西。

希望这篇文章，能帮你在 Monorepo 这条路上少走一点弯路。
